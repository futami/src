  47 /**

  48  * A subgraph is a set of items that are electrically connected on a single sheet.

  49  *

  50  * For example, a label connected to a wire and so on.

  51  * A net is composed of one or more subgraphs.

  52  *

  53  * A set of items that appears to be physically connected may actually be more

  54  * than one subgraph, because some items don't connect electrically.

  55  *

  56  * For example, multiple bus wires can come together at a junction but have

  57  * different labels on each branch.  Each label+wire branch is its own subgraph.

  58  *

  59  */
  60 class CONNECTION_SUBGRAPH
  61 {
  62 public:
  63     enum class PRIORITY
  64     {
  65         INVALID = -1,
  66         NONE    = 0,
  67         PIN,
  68         SHEET_PIN,
  69         HIER_LABEL,
  70         LOCAL_LABEL,
  71         POWER_PIN,
  72         GLOBAL
  73     };

  76               m_graph( aGraph ),

 345     /**

 346      * Updates the graphical connectivity between items (i.e. where they touch)

 347      * The items passed in must be on the same sheet.

 348      *

 349      * In the first phase, all items in aItemList have their connections

 350      * initialized for the given sheet (since they may have connections on more

 351      * than one sheet, and each needs to be calculated individually).  The

 352      * graphical connection points for the item are added to a map that stores

 353      * (x, y) -> [list of items].

 354      *

 355      * Any item that is stored in the list of items that have a connection point

 356      * at a given (x, y) location will eventually be electrically connected.

 357      * This means that we can't store SCH_COMPONENTs in this map -- we must store

 358      * a structure that links a specific pin on a component back to that

 359      * component: a SCH_PIN_CONNECTION.  This wrapper class is a convenience for

 360      * linking a pin and component to a specific (x, y) point.

 361      *

 362      * In the second phase, we iterate over each value in the map, which is a

 363      * vector of items that have overlapping connection points.  After some

 364      * checks to ensure that the items should actually connect, the items are

 365      * linked together using ConnectedItems().

 366      *

 367      * As a side effect, items are loaded into m_items for BuildConnectionGraph()

 368      *

 369      * @param aSheet is the path to the sheet of all items in the list

 370      * @param aItemList is a list of items to consider

 371      */
 372     void updateItemConnectivity( const SCH_SHEET_PATH& aSheet,
 373                                  const std::vector<SCH_ITEM*>& aItemList );
 374 
 375     /**

 376      * Generates the connection graph (after all item connectivity has been updated)

 377      *

 378      * In the first phase, the algorithm iterates over all items, and then over

 379      * all items that are connected (graphically) to each item, placing them into

 380      * CONNECTION_SUBGRAPHs.  Items that can potentially drive connectivity (i.e.

 381      * labels, pins, etc.) are added to the m_drivers vector of the subgraph.

 382      *

 383      * In the second phase, each subgraph is resolved.  To resolve a subgraph,

 384      * the driver is first selected by CONNECTION_SUBGRAPH::ResolveDrivers(),

 385      * and then the connection for the chosen driver is propagated to all the

 386      * other items in the subgraph.

 387      */
 388     void buildConnectionGraph();
